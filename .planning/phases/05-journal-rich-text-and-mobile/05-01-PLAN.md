**DayStrip (MOB-07):** The root cause is that `dateStr` uses `getLocalDateString(d, dayStartHour)` (offset-aware) but `dayAbbr` and `dateNum` are derived from the raw `Date` object `d` (calendar date). Fix by computing the "logical today" first, then building all cells backward from it:

1. Compute `logicalToday = getLocalDateString(new Date(), dayStartHour)` at the top of the cell-generation logic
2. For each cell index `i` (counting backward from logicalToday):
   - Parse logicalToday into year/month/day parts: `const [y, m, d] = logicalToday.split('-').map(Number)`
   - Create `base = new Date(y, m - 1, d)` then `base.setDate(base.getDate() - i)`
   - Derive `dateStr = new Intl.DateTimeFormat('en-CA').format(base)` (YYYY-MM-DD)
   - Derive `dayAbbr = new Intl.DateTimeFormat('en-US', { weekday: 'short' }).format(base)`
   - Derive `dateNum = base.getDate()`
3. This ensures labels and dateStr always agree because both derive from the same offset-aware base

**ConsistencyGraph (MOB-08):** In the totalCompleted accumulation loop, replace the current `cell.entry.completed` access with a NaN-safe version:
```tsx
if (cell.entry) {
  if (typeof cell.entry === 'boolean') {
    totalCompleted += cell.entry ? 1 : 0;
  } else {
    totalCompleted += Number(cell.entry.completed) || 0;
  }
}
```
Also ensure the heatmap cell date generation uses the same offset-aware pattern as the DayStrip fix — dates passed into the completionMap lookup must use `getLocalDateString` with `dayStartHour`.
