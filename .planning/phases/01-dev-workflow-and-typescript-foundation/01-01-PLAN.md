  <action>
1. Replace `tsconfig.json` with project references root:
```json
{
  "files": [],
  "references": [
    { "path": "./tsconfig.app.json" },
    { "path": "./tsconfig.node.json" }
  ]
}
```

2. Create `tsconfig.app.json`:
```json
{
  "compilerOptions": {
    "target": "ES2020",
    "lib": ["ES2020", "DOM", "DOM.Iterable"],
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "allowJs": true,
    "strict": false,
    "skipLibCheck": true,
    "isolatedModules": true,
    "noEmit": true,
    "resolveJsonModule": true,
    "allowImportingTsExtensions": true,
    "types": ["vitest/globals"],
    "paths": { "@/*": ["./src/*"] },
    "baseUrl": "."
  },
  "include": ["src"]
}
```
Note: `allowJs: true` lets .js/.jsx coexist during migration. `strict: false` avoids error cascade before migration. `types: ["vitest/globals"]` prevents TS errors on describe/it/expect in test files.

3. Create `tsconfig.node.json`:
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2023"],
    "module": "ESNext",
    "moduleResolution": "bundler",
    "skipLibCheck": true,
    "noEmit": true,
    "isolatedModules": true,
    "allowImportingTsExtensions": true
  },
  "include": ["vite.config.ts", "vitest.config.ts"]
}
```
Note: include references the future .ts names. This is fine -- tsconfig.node.json won't be used by tsc until those files are renamed in Plan 03.

4. Create `src/vite-env.d.ts`:
```typescript
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_USER_ID: string
  readonly VITE_USER_JWT: string
  readonly VITE_SUPABASE_URL: string
  readonly VITE_SUPABASE_ANON_KEY: string
  readonly VITE_ENABLE_DEV_TOOLS?: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}

declare const __DEV_TOOLS_ENABLED__: boolean
```

5. Generate Supabase database types:
```bash
supabase gen types typescript --linked --schema public > src/lib/database.types.ts
```
If the supabase CLI requires login or the linked project is unavailable, create a placeholder file with a comment explaining how to regenerate, and a minimal `export type Database = Record<string, unknown>` so imports don't break.

6. Update `vite.config.js` to add the `define` block for dev tools gating. Add this inside `defineConfig({...})`:
```javascript
define: {
  __DEV_TOOLS_ENABLED__: JSON.stringify(
    process.env.VITE_ENABLE_DEV_TOOLS === 'true' || process.env.NODE_ENV !== 'production'
  ),
},
```

7. Create/update `.env` (NOT `.env.local` -- this is the non-secret defaults file) to include:
```
VITE_ENABLE_DEV_TOOLS=true
```
This ensures local development always shows dev tools. On Vercel production (main branch), this env var will not be set, so dev tools will be stripped.

8. Update AppShell.jsx: Find the existing dev tools conditional render (likely using `import.meta.env.DEV`) and change it to use `__DEV_TOOLS_ENABLED__` instead. For example, change:
```jsx
{import.meta.env.DEV && <DevToolsPanel ... />}
```
to:
```jsx
{__DEV_TOOLS_ENABLED__ && <DevToolsPanel ... />}
```
Also update the Ctrl+Shift+D keyboard listener conditional to use `__DEV_TOOLS_ENABLED__`.

Update `eslint.config.js` to cover TypeScript files:

1. Change the `files` glob from `['**/*.{js,jsx}']` to `['**/*.{js,jsx,ts,tsx}']`.

2. That's the minimal change needed. The existing ESLint plugins (react-hooks, react-refresh) work on .ts/.tsx files without additional parser config because ESLint's flat config with `ecmaFeatures: { jsx: true }` handles TSX.

Note: We are NOT adding `@typescript-eslint/parser` or `@typescript-eslint/eslint-plugin` in this phase. TypeScript's own `tsc --noEmit` handles type checking. Adding TS-ESLint would be a separate concern and would significantly increase config complexity. The current setup (ESLint for React rules + tsc for types) is the standard Vite approach.
