# Phase 05: Push Notifications - Research

**Researched:** 2026-03-14
**Domain:** Web Push API, Service Workers, Supabase Edge Functions, pg_cron scheduling
**Confidence:** MEDIUM

## Summary

This phase replaces the existing notification stubs (`src/lib/notifications.js`) with real Web Push API delivery. The architecture has three layers: (1) a service worker in the browser that listens for push events and displays notifications, (2) client-side subscription management that registers with the Push API and stores the subscription in Supabase, and (3) a server-side trigger (Supabase Edge Function invoked by pg_cron) that sends push messages at the user's configured morning/evening times.

The key constraint is that this is a single-user app deployed on Vercel's Hobby tier. Vercel cron jobs are limited to once-per-day on the free tier, making them unsuitable for twice-daily notifications. Supabase pg_cron + Edge Functions is the correct path -- pg_cron runs inside Postgres on any Supabase plan (including free), and can schedule at minute-level granularity. The Edge Function receives the cron call and sends the web push message using VAPID authentication.

The existing `user_settings` table already stores `morning_prompt_hour` and `evening_prompt_hour`, and the `settingsStore` caches these client-side. The notification times are already configurable -- this phase wires them to actual delivery.

**Primary recommendation:** Use `vite-plugin-pwa` with `injectManifest` strategy for the service worker, store push subscriptions in a new Supabase table, and use a Supabase Edge Function triggered by pg_cron to send notifications via the Web Push protocol with VAPID keys.

### Pattern 1: Service Worker with Push Event Listener
**What:** Custom SW using injectManifest that handles both precaching and push events
**When to use:** Always -- this is the core of push notification delivery
**Example:**
```javascript
// src/sw.js
import { precacheAndRoute, cleanupOutdatedCaches } from 'workbox-precaching';
import { clientsClaim } from 'workbox-core';

cleanupOutdatedCaches();
precacheAndRoute(self.__WB_MANIFEST);
self.skipWaiting();
clientsClaim();

// Push notification handler
self.addEventListener('push', (event) => {
  const data = event.data?.json() ?? {};
  const title = data.title || 'wintrack';
  const options = {
    body: data.body || 'Time to check in!',
    icon: '/logo.png',
    badge: '/logo.png',
    tag: data.tag || 'wintrack-reminder', // prevents duplicate notifications
    renotify: true,
  };
  event.waitUntil(self.registration.showNotification(title, options));
});

// Notification click handler -- open app
self.addEventListener('notificationclick', (event) => {
  event.notification.close();
  event.waitUntil(
    clients.matchAll({ type: 'window', includeUncontrolled: true }).then((windowClients) => {
      if (windowClients.length > 0) {
        return windowClients[0].focus();
      }
      return clients.openWindow('/');
    })
  );
});
```

### Pattern 2: Client-side Push Subscription
**What:** Subscribe browser to push service, store subscription in Supabase
**When to use:** After user grants notification permission
**Example:**
```javascript
// src/lib/push-subscription.js
import { supabase } from '@/lib/supabase';
import { USER_ID } from '@/lib/env';

const VAPID_PUBLIC_KEY = import.meta.env.VITE_VAPID_PUBLIC_KEY;

function urlBase64ToUint8Array(base64String) {
  const padding = '='.repeat((4 - base64String.length % 4) % 4);
  const base64 = (base64String + padding).replace(/-/g, '+').replace(/_/g, '/');
  const rawData = atob(base64);
  return Uint8Array.from([...rawData].map((c) => c.charCodeAt(0)));
}

export async function subscribeToPush() {
  const registration = await navigator.serviceWorker.ready;
  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY),
  });

  const json = subscription.toJSON();
  await supabase.from('push_subscriptions').upsert({
    user_id: USER_ID,
    endpoint: json.endpoint,
    p256dh: json.keys.p256dh,
    auth: json.keys.auth,
    expiration_time: json.expirationTime,
  }, { onConflict: 'user_id' });

  return subscription;
}

export async function unsubscribeFromPush() {
  const registration = await navigator.serviceWorker.ready;
  const subscription = await registration.pushManager.getSubscription();
  if (subscription) {
    await subscription.unsubscribe();
    await supabase.from('push_subscriptions').delete().eq('user_id', USER_ID);
  }
}
```

### Pattern 3: Supabase Edge Function for Sending Push
**What:** Deno Edge Function that reads subscriptions and sends web push messages
**When to use:** Called by pg_cron at scheduled notification times
**Example:**
```typescript
// supabase/functions/send-push/index.ts
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';
import * as webpush from 'jsr:@negrel/webpush';

const vapidKeys = {
  publicKey: Deno.env.get('VAPID_PUBLIC_KEY')!,
  privateKey: Deno.env.get('VAPID_PRIVATE_KEY')!,
};

Deno.serve(async (req) => {
  const { type } = await req.json(); // 'morning' or 'evening'

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!,
  );

  // Get all subscriptions
  const { data: subs } = await supabase.from('push_subscriptions').select('*');
  if (!subs?.length) return new Response('No subscriptions', { status: 200 });

  const messages = {
    morning: { title: 'wintrack', body: "What's the grind for today?" },
    evening: { title: 'wintrack', body: 'Time for your evening check-in' },
  };

  const appServer = await webpush.ApplicationServer.new({
    contactInformation: 'mailto:admin@example.com',
    vapidKeys,
  });

  for (const sub of subs) {
    const subscriber = appServer.subscribe({
      endpoint: sub.endpoint,
      keys: { p256dh: sub.p256dh, auth: sub.auth },
    });
    await subscriber.pushTextMessage(JSON.stringify(messages[type] || messages.morning), {});
  }

  return new Response('Sent', { status: 200 });
});
```

### Pattern 4: pg_cron Scheduling
**What:** Postgres cron jobs that call the Edge Function at user-configured times
**When to use:** Set up once during deployment; update when user changes notification hours
**Example:**
```sql
-- Store secrets in Vault
SELECT vault.create_secret('https://PROJECT.supabase.co', 'project_url');
SELECT vault.create_secret('YOUR_ANON_KEY', 'anon_key');

-- Morning notification at configured hour (default 9am)
SELECT cron.schedule(
  'morning-push',
  '0 9 * * *',
  $$
  SELECT net.http_post(
    url := (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'project_url')
           || '/functions/v1/send-push',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'anon_key')
    ),
    body := '{"type":"morning"}'::jsonb
  ) AS request_id;
  $$
);

-- Evening notification at configured hour (default 9pm)
SELECT cron.schedule(
  'evening-push',
  '0 21 * * *',
  $$
  SELECT net.http_post(
    url := (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'project_url')
           || '/functions/v1/send-push',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'anon_key')
    ),
    body := '{"type":"evening"}'::jsonb
  ) AS request_id;
  $$
);
```

### Anti-Patterns to Avoid
- **Client-side setTimeout for notification scheduling:** Tab may be closed; push must come from server
- **Storing VAPID private key in client code:** Private key is server-only; only public key goes to client via env var
- **Using Notification API directly without service worker:** Direct `new Notification()` only works when tab is open; SW-based push works when app is closed
- **Skipping the `tag` option on showNotification:** Without tag, duplicate notifications stack up if cron fires while one is still showing

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Service worker build pipeline | Manual SW bundling | vite-plugin-pwa injectManifest | Handles Workbox manifest injection, TypeScript compilation, dev/prod modes |
| Web Push protocol encryption | Custom ECDH + HKDF | @negrel/webpush (Deno) or web-push (Node) | RFC 8291 encryption is complex; libraries handle padding, key exchange, payload encryption |
| VAPID key generation | Manual openssl commands | `npx web-push generate-vapid-keys` | Generates correctly formatted ECDSA P-256 keys |
| Base64url encoding | Custom encoder | Built-in `urlBase64ToUint8Array` helper | Standard pattern from MDN docs; tricky padding/charset edge cases |
| Cron scheduling infrastructure | Custom scheduler process | Supabase pg_cron | Already running in Postgres; zero additional infrastructure |

**Key insight:** The Web Push protocol involves RFC 8188 content encryption, ECDH key agreement, and HKDF key derivation. This is not something to implement manually. Libraries exist for every runtime (Node, Deno, Go, etc.).

## Common Pitfalls

### Pitfall 1: Notification Permission Timing
**What goes wrong:** Requesting notification permission immediately on page load, causing the browser to block the request
**Why it happens:** Browsers require a user gesture (click/tap) before `Notification.requestPermission()` will show the prompt; some browsers silently deny if called without a gesture
**How to avoid:** Show a custom in-app prompt first ("Enable notifications?") and only call `requestPermission()` inside the click handler for that prompt
**Warning signs:** `Notification.permission` stays `'default'` after calling requestPermission

### Pitfall 2: Service Worker Scope
**What goes wrong:** SW registered at wrong scope can't intercept push events for the app
**Why it happens:** SW scope defaults to the directory of the SW file; if placed in a subdirectory, it won't cover the root
**How to avoid:** vite-plugin-pwa handles this automatically by placing sw.js at root scope `/`
**Warning signs:** `navigator.serviceWorker.ready` never resolves; push subscription fails

### Pitfall 3: VAPID Key Format Mismatch
**What goes wrong:** `applicationServerKey` rejected by PushManager.subscribe()
**Why it happens:** The key must be a Uint8Array, but VAPID public key is base64url-encoded; wrong encoding (standard base64 vs base64url) causes `InvalidCharacterError`
**How to avoid:** Always use the `urlBase64ToUint8Array()` helper function with correct padding and charset conversion
**Warning signs:** `DOMException: Failed to execute 'subscribe'` with key-related message

### Pitfall 4: pg_cron Timezone
**What goes wrong:** Notifications fire at wrong time
**Why it happens:** pg_cron uses the Postgres server timezone (typically UTC); user's configured hours are in their local timezone
**How to avoid:** Either (a) store the user's timezone in settings and convert in the Edge Function, or (b) since this is a single-user app, set the cron schedule to the UTC equivalent of the user's local time. Option (b) is simpler -- when user updates settings, update the cron schedule
**Warning signs:** Notifications arriving at wrong hour; off by timezone offset

### Pitfall 5: Subscription Expiration
**What goes wrong:** Push messages stop arriving after weeks/months
**Why it happens:** PushSubscription can expire (browser may invalidate it); the `expirationTime` field may be null (no expiry) or a timestamp
**How to avoid:** Re-subscribe on every app load (idempotent if subscription hasn't changed), and handle 410 Gone responses in the Edge Function by deleting stale subscriptions
**Warning signs:** Edge Function gets 410 status from push endpoint

### Pitfall 6: Vercel Cron Limitation
**What goes wrong:** Deploying cron-based notification sending on Vercel Hobby fails
**Why it happens:** Vercel Hobby plan limits cron to once per day; morning + evening requires at minimum twice daily
**How to avoid:** Use Supabase pg_cron instead -- it has no frequency restrictions on any plan
**Warning signs:** Vercel deployment error about cron frequency

### Database Migration for Push Subscriptions
```sql
-- supabase/migrations/006_push_subscriptions.sql
CREATE TABLE push_subscriptions (
  user_id    uuid PRIMARY KEY DEFAULT auth.uid(),
  endpoint   text NOT NULL,
  p256dh     text NOT NULL,
  auth       text NOT NULL,
  expiration_time timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

ALTER TABLE push_subscriptions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can read own subscription"
  ON push_subscriptions FOR SELECT
  USING (user_id = (current_setting('request.jwt.claims', true)::json->>'sub')::uuid);

CREATE POLICY "Users can insert own subscription"
  ON push_subscriptions FOR INSERT
  WITH CHECK (user_id = (current_setting('request.jwt.claims', true)::json->>'sub')::uuid);

CREATE POLICY "Users can update own subscription"
  ON push_subscriptions FOR UPDATE
  USING (user_id = (current_setting('request.jwt.claims', true)::json->>'sub')::uuid)
  WITH CHECK (user_id = (current_setting('request.jwt.claims', true)::json->>'sub')::uuid);

CREATE POLICY "Users can delete own subscription"
  ON push_subscriptions FOR DELETE
  USING (user_id = (current_setting('request.jwt.claims', true)::json->>'sub')::uuid);
```

### Notification Permission UI Component
```jsx
// src/components/NotificationPermission.jsx
import { useState, useEffect } from 'react';
import { Bell, BellOff } from 'lucide-react';
import { subscribeToPush, unsubscribeFromPush } from '@/lib/push-subscription';

export function NotificationPermission() {
  const [permission, setPermission] = useState(
    typeof Notification !== 'undefined' ? Notification.permission : 'denied'
  );
  const [subscribed, setSubscribed] = useState(false);

  useEffect(() => {
    // Check existing subscription on mount
    navigator.serviceWorker?.ready.then((reg) => {
      reg.pushManager.getSubscription().then((sub) => {
        setSubscribed(!!sub);
      });
    });
  }, []);

  async function handleEnable() {
    const result = await Notification.requestPermission();
    setPermission(result);
    if (result === 'granted') {
      await subscribeToPush();
      setSubscribed(true);
    }
  }

  async function handleDisable() {
    await unsubscribeFromPush();
    setSubscribed(false);
  }

  if (permission === 'denied') return null; // Browser blocked, nothing we can do

  return subscribed ? (
    <button onClick={handleDisable} className="flex items-center gap-2">
      <Bell className="h-4 w-4" /> Notifications on
    </button>
  ) : (
    <button onClick={handleEnable} className="flex items-center gap-2">
      <BellOff className="h-4 w-4" /> Enable notifications
    </button>
  );
}
```

## Open Questions

1. **pg_cron timezone handling for dynamic schedule updates**
   - What we know: pg_cron runs in Postgres server timezone (UTC on Supabase hosted)
   - What's unclear: Best pattern for updating cron schedule when user changes notification hours in settings. Options: (a) Edge Function reads settings and decides whether to send, (b) update cron.schedule SQL when settings change
   - Recommendation: Option (a) is simpler -- keep cron running hourly (or at a few candidate hours), and let the Edge Function read user_settings to decide whether to actually send. This avoids needing client-side SQL to modify cron jobs.

2. **PWA icon sizes**
   - What we know: Only `logo.png` exists in public/ currently
   - What's unclear: Whether additional icon sizes (192x192, 512x512) are needed for the manifest
   - Recommendation: Generate standard PWA icon sizes from existing logo during implementation

3. **@negrel/webpush compatibility with Supabase Edge Functions**
   - What we know: It's a JSR package built for Deno using only Web APIs
   - What's unclear: Whether the latest version works with Supabase's Deno runtime version without issues
   - Recommendation: Test during implementation; fallback is manual fetch to push endpoint with web-push protocol headers (more work but no external dependency)
