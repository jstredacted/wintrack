```jsx
// Bug 1 (line 49): style={{ animation: open ? 'overlay-enter 0.3s ease forwards' : undefined }}
//   'overlay-enter' is a CSS class name, NOT a @keyframes name

// Bug 2 (line 10-12): useEffect only handles open=true (setVisible(true))
//   When open goes false, nothing calls setExiting(true) — so exiting is never true

// (no exiting state)

  if (open) setVisible(true);  // only handles open=true

```

Canonical state-machine pattern (from WinInputOverlay.jsx — the reference implementation):
```jsx
const [visible, setVisible] = useState(false);
const [exiting, setExiting] = useState(false);

useEffect(() => {
  if (open) { setVisible(true); setExiting(false); }
  else if (visible) { setExiting(true); }
}, [open]);

// In JSX:
className={`fixed inset-0 ... ${exiting ? 'overlay-exit' : 'overlay-enter'}`}
onAnimationEnd={(e) => {
  if (e.target !== e.currentTarget) return;
  if (exiting) setVisible(false);
}}
```

CSS classes in index.css (confirmed @keyframes names):
- `.overlay-enter { animation: overlay-slide-in 0.3s ... }`  ← class applies the named keyframe
- `.overlay-exit { animation: overlay-slide-out 0.3s ... }` ← class applies the named keyframe
- `@keyframes overlay-slide-in` — actual keyframe definition
- `@keyframes overlay-slide-out` — actual keyframe definition

StreakCelebration has no child motion components, so no bubble guard is needed on onAnimationEnd.

The `displayCount` animation (requestAnimationFrame ramp) and the auto-close setTimeout both reset correctly when `open` changes — no changes needed to those effects.

The `if (!open) { setDisplayCount(0); return; }` in the count animation effect works correctly with the new pattern: when open goes false, exiting=true starts the exit, then after animation completes visible=false and the component unmounts. The displayCount reset on open=false is fine.
