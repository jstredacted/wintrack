<objective>
Create the server-side push notification delivery system: a Supabase Edge Function that sends web push messages to stored subscriptions, and pg_cron SQL scripts to schedule hourly checks.

Purpose: Push notifications must arrive even when the app tab is closed. This requires server-side triggering. The Edge Function reads subscriptions from the DB and sends via VAPID-authenticated Web Push protocol. pg_cron calls the function hourly; the function itself decides whether to send based on user_settings morning/evening hours.

Output: Deployable Edge Function, pg_cron setup SQL script, human verification of end-to-end flow.
</objective>

  <action>
1. Create directory `supabase/functions/send-push/` if it doesn't exist.

2. Create `supabase/functions/send-push/index.ts`:
   - Import createClient from 'https://esm.sh/@supabase/supabase-js@2'
   - Import webpush from 'jsr:@negrel/webpush'
   - Read env: VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY
   - Deno.serve handler:
     a. Parse request JSON for `{ type }` — 'morning', 'evening', or 'check' (hourly)
     b. Create Supabase client with service role key
     c. Read user_settings to get morningPromptHour and eveningPromptHour: `supabase.from('user_settings').select('morning_prompt_hour, evening_prompt_hour').single()`
     d. Determine current hour in user's timezone. Since this is a single-user app, use a simple approach: the Edge Function receives the hour to check from pg_cron, OR reads current UTC hour. Simplest: pg_cron fires hourly, function checks if current UTC hour matches either setting. **Important:** For the single-user case, store user's timezone offset in user_settings OR accept that the user sets morning_prompt_hour/evening_prompt_hour in UTC-equivalent terms. The simplest approach per research: pg_cron fires hourly, Edge Function reads settings and compares against current UTC hour.
     e. If type is 'check': get current UTC hour, compare against morning_prompt_hour and evening_prompt_hour from settings. If no match, return early with "Not notification time".
     f. If type is 'morning' or 'evening' (direct invoke for testing): skip the hour check, proceed to send.
     g. Query `push_subscriptions` for all subscriptions
     h. If no subscriptions, return 200 "No subscriptions"
     i. Build message: morning = "What's the grind for today?", evening = "Time for your evening check-in"
     j. Create ApplicationServer with VAPID keys and `contactInformation: 'mailto:admin@wintrack.app'`
     k. For each subscription: create subscriber, call `pushTextMessage(JSON.stringify({ title: 'wintrack', body: message, tag: 'wintrack-reminder' }), {})`
     l. Handle 410 Gone responses by deleting stale subscriptions from DB
     m. Return 200 with count of sent messages
   - Add CORS headers for Supabase invocation

3. Create `scripts/setup-push-cron.sql`:
   - Comment header explaining this script must be run in Supabase SQL Editor
   - Enable pg_cron and pg_net extensions (CREATE EXTENSION IF NOT EXISTS)
   - Store project URL and anon key in vault:
     ```sql
     SELECT vault.create_secret('https://YOUR_PROJECT.supabase.co', 'project_url');
     SELECT vault.create_secret('YOUR_ANON_KEY', 'anon_key');
     ```
     (With comments to replace placeholders)
   - Schedule hourly cron job that calls the Edge Function with type 'check':
     ```sql
     SELECT cron.schedule(
       'push-notification-check',
       '0 * * * *',  -- Every hour on the hour
       $$ SELECT net.http_post(...) $$
     );
     ```
   - The hourly approach means the Edge Function decides whether to send based on user_settings — no need to update cron when settings change
   - Include a helper to view cron job status: `SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;`
